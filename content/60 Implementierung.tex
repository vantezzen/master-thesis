\section{Implementierung des Constraint-Systems}\label{section:Implementierung des Constraint-Systems}

Das Constraint-System baut, wie in der Konzeptionalisierung aufgezeigt, auf den Ergebnissen der vorherigen Forschungsarbeiten auf.

Als Basis der Codebasis wird das JavaScript Build-System \textit{Vite} genutzt \parencite{voidzeroinc.Vite2025}. Das System automatisiert den für moderne Webanwendungen nötigen Übersetzungs- und Optimierungsprozess, bei welchem der Quelltext in TypeScript in eine für Browser interpretierbare Ausgabe umgewandelt wird. Vite löst hierfür Modulabhängigkeiten auf und bündelt statische Ressourcen, wodurch ein lauffähiges Artefakt entsteht.

Über den reinen Transpilationsschritt hinaus führt Vite in einem Produktionsbuild zusätzliche Optimierungen – etwa Code-Splitting, Tree-Shaking und Minifizierung – durch, um Ladezeiten und Bandbreitenbedarf zu minimieren.

\medskip

Wie in im Kapitel \ref{section:Technische Grundlagen} [S. \pageref{section:Technische Grundlagen}] zu technischen Grundlagen bereits aufgearbeitet, wird als Basis für die Darstellung React, react-three-fiber sowie die Erweiterung @react-three/xr genutzt.

Der Code für React-Anwendungen geschieht durch einen funktionalen Codestil, der Rest des Validierungssystem soll jedoch über eine objektorientierte Architektur umgesetzt werden.

\subsection{Aufbau des Codes}

Die Codebasis wurde in drei Haupteinheiten geteilt: Der Application Layer stellt den Kern des Validierungssystem und ist für das Parsing und Auswerten von Validierungen zuständig. Der Infrastructure Layer stellt Hilfssysteme für die Auswertung, darunter etwa zur Segmentierung der Umgebung. Zuletzt ist der Presentation Layer für die Darstellung der 3D Objekte und Validierungsergebnisse über React-Three-Fiber zuständig.

In der ersten Iteration wurde die Kommunikation zwischen Einheiten der Application und Infrastructure zunächst über ein \quoted{God Object} umgesetzt. Um dieses Anti-Pattern aufzulösen wurde diese Abhängigkeit durch die Nutzung von Dependency Injection mittels der TypeScript Bibliothek \textmono{tsyringe} umgesetzt - hierdurch werden alle Abhängigkeiten über einen einheitlichen IoC-Container aufgelöst werden. Zusätzlich können durch den Container Referenzen auf Interfaces oder abstrakte Klassen durch die konkret benötigten Implementierungen umgewandelt werden.

\subsubsection{Kernsystem: Application Layer}\label{subsection:Kernsystem}

\begin{figure}[h]
\includegraphics[width=1\textwidth]{abb/uml core system.png}
\centering
\caption{Klassendiagramm: Application Layer}
\label{fig:uml core system}
\end{figure}

Abbildung \ref{fig:uml core system} zeigt ein UML Klassendiagramm über den Aufbau des Kernsystems. Das oberste Objekt des Diagramms ist die Klasse \textmono{ValidationOrchestrator}, welche die Deligierung der Aufgaben an die einzelnen Subklassen unternimmt und für den Presentation Layer eine einheitliche API bereitstellt. Hierfür existiert die vorher konzeptionierte Funktion \textmono{validate(object: Object): Promise<ValidationResult>} zur Validierung.

Wie zuvor aufgezeigt erfolgt die Auflösung der Verbindungen zu den Subklassen über den Dependency Injection IoC-Container, um die einzelnen Aufgaben weiterzugeben.

Zur zentralen Verwaltung aller Objekte wurde eine Klasse \textmono{Object} umgesetzt. Diese Klasse enthält alle Informationen zum Objekttyp, Position, Skalierung, Rotation sowie einer Objekt ID zur eindeutigen Identifizierung einzelner Objekte im System. Zusätzlich setzt die Klasse das Observer Pattern durch die Implementierung eines \textmono{EventEmitters} um, wodurch alle Komponenten des Systems bei Veränderungen informiert werden.

\medskip

Zunächst ruft der Delegator daraufhin die Klasse \textmono{ValidationRuleResolver} mit der Funktion \textmono{resolveRulesetForObject} auf. Diese Klasse lädt alle Constraint-Rulset Dateien, um die objektspezifischen Regeln über die interne Methode \textmono{resolveDirectRulesetsForObject} sowie global anwendbaren Regeln über die interne Methode \textmono{resolveGlobalRulesForTags} aufzulösen. Alle anwendbaren Regeln werden schließlich zu einem \textmono{ResolvedRuleset} verbunden, welches von späteren Einheiten ausgeführt werden kann.

Der \textmono{ValidationOrchestrator} ruft daraufhin die Methode \textmono{execute(object: Object, rules: ResolvedRuleset)} des \textmono{ValidationExecutor} auf. Innerhalb dieser Methode werden die benötigten Validatoren erstellt und ausgeführt.

\medskip

Die tatsächliche Ausführung der Logik erfolgt über Validatoren, welche auf der abtrakten Klasse \textmono{Validator<RuleType>} aufbauen. Die abstrakte Klasse implementiert bereits einige Hilfsmethoden, um die Zuständigkeit für einzelne Regeln zu prüfen, das Schema der Regeln zu validieren sowie die Art der Regeln (\textmono{forbid} oder \textmono{allow-only}) auszuwerten. Die konkreten Implementierungen werden später genauer besprochen, diese müssen jedoch nur noch die Methode \textmono{passes} sowie konkrete Regel-Validierungen implementieren, um einen positiv-Test auf den Constraint durchzuführen.

Für die Erstellung der Validatoren wird das Factory Pattern angwendet, bei welchem die Factory-Funktion \textmono{createValidators} Instanzen aller Validatoren über den IoC-Container auflöst.

\medskip

Durch die einzelnen Validatoren sowie den \textmono{ValidationExecutor} werden schließlich ein Array an \textmono{ValidationCheckResult} zurückgemeldet, welche Informationen über den Fehlerstatus sowie betroffene Bereiche der Szene liefern.

Diese Information wird schließlich vom \textmono{ValidationOrchestrator} an die Subklasse \textmono{ValidationReporter} deligiert, welcher die einzelnen Fehler in zusammenhängende Fehlermeldungen zusammenfasst und über den \textmono{ErrorMessageService} formatiert.

\medskip

Ein zentraler Aspekt des Containt-Syntax sind \quoted{atomare Fehlermeldungen}, welche modular zu einem zusammenhängenden Satz kombiniert werden können. Die atomaten Fehler müssen hierbei im Format \quoted{Das Objekt muss [Fehlermeldung], damit [Grund]} formatiert werden, z.B. \quoted{1 Meter von einer Straße entfernt liegen, damit die Gefahr bei Unfällen minimiert wird} und \quoted{vollständig auf Gras platziert werden, damit es keine Fußgänger behindert}. Über den \textmono{ErrorMessageService} können diese einzelnen atomaren Fehler schließlich zu einer vollständigen Information zusammengefügt werden, z.B. \quoted{Die Sitzbank muss \textit{1 Meter von einer Straße entfernt liegen, damit die Gefahr bei Unfällen minimiert wird} und \textit{vollständig auf Gras platziert werden, damit es keine Fußgänger behindert}}.

Die vollständigen, kombinierten Fehlermeldungen werden schließlich zurück an den Presentation Layer gemeldet, damit diese dort dargestellt werden können.

\subsubsection{Service-System: Infrastructure Layer}

Der Infrastructure Layer stellt diverse geteilte Services zur Verfügung, welche in den Validierungen gebraucht werden.

\begin{figure}[h]
\includegraphics[width=1\textwidth]{abb/uml infrastructure.png}
\centering
\caption{Klassendiagramm: Infrastructure Layer}
\label{fig:uml infrastructure}
\end{figure}

Abbildung \ref{fig:uml infrastructure} zeigt ein UML Klassendiagramm des Aufbaus der Layers, wobei die einzelnen Services innerhalb der Validatoren mittels des IoC ServiceContainers aufgelöst werden.

Der \textmono{GroundService} stellt die Methode \textmono{getGroundType} zur Bestimmung, welche Untergrundarten sich unter einem Objekt befindet. Intern prüft diese Funktion, welcher Untergrund sich an den vier äußersten Ecken des Objekts befindet. Zur Berechnung der Ecken nutzt der Service daher den \textmono{SizeService}.

Der \textmono{SizeService} stellt die Methode \textmono{getObjectSize} zur Berechnung der Eckpunkte eines Objekts anhand der Objektmodels, der eingestellten Größe, Position und Rotation. Aufgrund der Abhängigkeit des konkreten Objektmodels lädt und cached diese Klasse intern die Objektmodelle im Gltf-Format.

Der \textmono{SegmentationProvider} stellt schließlich eine Möglichkeit, auf die Segmentierungsinformationen an bestimmten Punkten des 3D-Raums zuzugreifen. Da die Segmentierung, wie in der Konzeptionierung ausgearbeitet, auf diverse Arten umgesetzt werden kann, geschieht dies über das generische Interface \textmono{SegmentationProvider} und die konkrete Implementierung \textmono{GeoJsonSegmentationProvider}. Durch diese Separierung ist es in zukünftigen Iterationen leichter möglich, die Segmentierung durch andere Methoden wie AI-basierte Bilderkennung zu ersetzen.

\medskip

Insgesamt erlaubt diese Aufteilung in einen Infrastructure Layer, ein robustes Sharing von Logik, aber auch Caching-Informationen zwischen Objekt-Validierungen und Klassen.

\subsubsection{AR-Basissystem: Presentation Layer}

Das AR-Basissystem ist für die Darstellung und Interaktion von Objekten zuständig und zeigt generierte Validierungsergebnisse für den Nutzer an. Teile der grundlegenden Logik und Interaktionstechniken konnten aus der vorhergehenden Forschungsarbeit übernommen werden, ein Großteil des Systems wurde jedoch für die Anforderungen dieser Arbeit neu entwickelt. Die Behandlung der visuellen Ausgabe dieser Schicht erfolgt in Kapitel \ref{subsection:Virtuelle Straßenumgebung} [S. \pageref{subsection:Virtuelle Straßenumgebung}].

\begin{figure}[h]
\includegraphics[width=1\textwidth]{abb/uml presentation.png}
\centering
\caption{Klassendiagramm: Presentation Layer}
\label{fig:uml presentation}
\end{figure}

Abbildung \ref{fig:uml presentation} visualisiert den Aufbau der Komponenten des Systems. Jede Klasse verfolgt dabei das Single-Responsibility-Principle und übergibt Delegationen an ihre direkten Kind-Elemente. Im Folgenden werden die einzelnen Bausteine im Detail erläutert.

Startpunkt für React während des Render-Vorgangs ist die Komponente \textmono{AppCanvas}. Diese Komponente rendert grundlegende, globale UI-Elemente wie einen Button zum Starten der AR-Ansicht und eine Komponente zum Hinzufügen von Objekten in die Szene. Zusätzlich setzt die Komponente die 3D-Umgebung auf, über welche alle weiteren Elemente dargestellt werden, durch die Initialisierung des \textmono{<Canvas>}-Kontext von react-three-fiber. Eine weitere Logik enthält diese Komponente bewusst nicht – sie delegiert das eigentliche Scene-Graph-Management an \textmono{AppCanvasContent}.

\medskip

Innerhalb dieser 3D-Umgebung befindet sich vor allem die Subkomponente \textmono{AppCanvasContent} für das Szenenmanagement und kombiniert die einzelnen UI-Aspekte:

\begin{itemize}
  \item \textbf{Interaktionsfluss} über \textmono{InteractionConnector} (Eingabegeräte, Gesten)
  \item \textbf{Objektdarstellung} durch eine kollektive Instanziierung von \textmono{CanvasObject}
  \item \textbf{Umgebungslayer} über \textmono{SegmentationVisualization}  
\end{itemize}

\medskip

Die Subkomponente \textmono{InteractionConnector} sowie die intern genutzte \textmono{Interaction} Klasse bündelt sämtliche Eingabekanäle, um die low-level Events über Mausklicks oder Touchscreen-Interaktionen in deklarative Aktionen (\textmono{place}, \textmono{move}, \textmono{rotate}) umzuwandeln und auf das aktuelle Objekt anzuwenden.

Hierdurch bleiben Präsentationskomponenten input-agnostisch und werden passiv über den \textmono{InteractionConnector} aktualisiert. Durch die Nutzung des Observer Patterns innerhalb der genutzten \textmono{Object}-Klasse, werden bei Veränderungen automatisch alle Validierungssysteme informiert, ohne dass dies explizit durch den Connector geschehen muss.

\medskip

Die Komponente \textmono{SegmentationVisualization} visualisiert die von der Geosegmentierung vorbereiteten Bereiche von Straßen, Gehwegen und Grasflächen - siehe Kapitel \ref{subsection:Virtuelle Straßenumgebung} [S. \pageref{subsection:Virtuelle Straßenumgebung}]. Dies ist vor allem für rein virtuelle Testumgebungen sinnvoll und um die GeoJSON-Areale in der AR-Umgebung über den realen Untergrund zu überlagern.

\medskip

Über die Komponente \textmono{CanvasObject} wird ein einzelnes, logisch-fachliches Objekt sowie alle weiteren verbundenen Elemente in der Szene dargestellt. Zur Darstellung der gesamten Szene wird daher durch die \textmono{AppCanvasContent} Komponente ein separates \textmono{CanvasObject} für jedes Objekt erstellt.

Jedes Objekt besteht hierfür intern wiederum aus drei Teilkomponenten:

\begin{enumerate}
  \item das Rendering des eigentlichen Objekts über \textmono{EffectableGltf}  
  \item Darstellung der Fehlermeldungen aus der Validierung über \textmono{ValidationErrors}
  \item Darstellung betroffener Fehlerbereiche durch \textmono{HighlightAreas}
\end{enumerate}

\medskip

Eine der zentralen Komponenten der Präsentationsschicht ist \textmono{EffectableGltf}.  
Ziel dieser Klasse ist es, beliebige glTF-Modelle um visuelle Effekte – etwa eine rote Hinterlegung bei Validierungsfehlern oder eine weiße Kontur zur Hervorhebung des aktuell ausgewählten Objekts – zu erweitern.

Im Ökosystem von \textmono{react-three-fiber} existieren zwar mehrere Lösungen für Post-Processing  
(z.B. \textmono{@react-three/postprocessing}, \textmono{react-three-fiber effects}), diese Bibliotheken beschränken sich jedoch häufig auf primitive Objekte (Würfel, Kugeln) oder erfordern tiefes Eingreifen in den Three-JS-Rendergraph.

Bei komplexen glTF-Strukturen können diese daher meist nicht ohne Umwandlungen genutzt werden, da Effekte rekursiv auf sämtliche \textmono{Mesh}-Knoten angewendet werden müssen. Mehrere Effekte können zusätzlich oft nicht problemlos kombiniert werden, ohne Artefakte oder Overdraw zu erzeugen, oder das dynamische Zu- und Abschalten der Effekte kann Memory Leaks im GPU-Kontext hinterlassen.

Erste Versuche, die Effekte direkt innerhalb der Projektbasis zu implementieren, zeigten einen kontinuierlichen Anstieg der Komplexität, insbesondere beim gleichzeitigen Einsatz mehrerer Overlays und Outlines.  
Um die Wartbarkeit zu erhöhen und einen Wiederverwendungsnutzen für nachgelagerte Projekte zu schaffen, wurde die Logik daher in eine eigenständige Open-Source-Bibliothek \textmono{@vantezzen/effectable-gltf}\footnote{\url{https://www.npmjs.com/package/@vantezzen/effectable-gltf}} überführt. Die Bibliothek wurde seit ihrer Veröffentlichung zusätzlich bereits in die offizielle Liste von Community-Paketen der React-Three-Fiber Dokumentation aufgenommen \parencite{poimandresCommunityR3FComponents2025}.

Die Bibliothek versteht sich als Drop-in-Replacement für das in \textmono{@react-three/drei} enthaltene \textmono{Gltf}-Element zur Darstellung von glTF-Modellen.
Effekte werden deklarativ als Kindelemente hinzugefügt; die Bibliothek sorgt intern für Shader-Injection, Depth-Handling und das Aufräumen temporärer Materialien.  

\begin{lstlisting}[language=html,caption={Deklarative Effekte mit \textmono{EffectableGltf}},label={lst:effectable}]
<EffectableGltf src="/assets/tree.gltf">
  <OverlayEffect color="red" opacity={0.5} />
  <OutlineEffect color="white" thickness={0.012} />
</EffectableGltf>
\end{lstlisting}

Listing \ref{lst:effectable} bietet ein einfaches Beispiel für die Nutzung der Bibliothek, wobei das Objekt \textmono{tree.gltf} geladen wird und mit den Effekten \textmono{OverlayEffect} in Rot mit 5\% Transparenz sowie \textmono{OutlineEffect} in weiß versehen.

\textmono{EffectableGltf} ermöglicht performante Renderpfade ohne zusätzliche Frame-Kosten (REQ-NF-002) und verständliche visuelle Rückmeldungen für Endnutzerinnen und Endnutzer. Die Auslagerung als dediziertes Paket gewährleistet zugleich die Nachnutzbarkeit in künftigen Projekten und reduziert technischen Schuldaufbau im Hauptrepository.

\medskip

Zusammengefasst besteht die Delegationskette der UI aus folgenden Elementen:

\begin{enumerate}
  \item \textmono{AppCanvas} initiiert XR-Kontext, delegiert an \textmono{AppCanvasContent}
  \item \textmono{AppCanvasContent} verteilt State an \textmono{CanvasObject}, \textmono{SegmentationVisualization}, \textmono{InteractionConnector}
  \item \textmono{CanvasObject} rendert \textmono{EffectableGltf}, \textmono{ValidationErrors}, \textmono{HighlightAreas}
  \item \textmono{InteractionConnector} abstrahiert Eingaben für alle Child-Komponenten
  \item \textmono{SegmentationVisualization} zeigt globale Kontextdaten an
\end{enumerate}

Diese strikt hierarchische Aufteilung stellt sicher, dass UI-Ereignisse, Validierungslogik und Rendering sauber entkoppelt bleiben.  Gleichzeitig erlaubt das Konzept die einfache Erweiterung um zusätzliche Visualisierungs- oder Interaktionsmodule, ohne bestehende Komponenten zu modifizieren.

\subsubsection{State-System}

Zur geteilte Verwaltung der aktuellen Objektdaten sowie globalen Zugriff auf die 3D-Szene wird die Bibliothek \textmono{zustand}\footnote{\url{http://npmjs.com/package/zustand}} genutzt. \textmono{zustand} ist ein leichtgewichtiges Zustandssystem für React, welches dazu dienen gemeinsam genutzte Zustände (z.B. alle Objekt-Klassen sowie das aktuell ausgewähltes Objekt) zentral zu verwalten und effizient an beteiligte Komponenten weiterzugeben, sowie Komponenten zu aktualisieren, wenn sich der Status ändert. Dieses Prinzip ist vergleichbar mit statischen Zustandscontainern oder Singleton-Instanzen in anderen Sprachen wie Java, mit dem Unterschied, dass \textit{zustand} sich direkt in die Reaktivität von React einfügt. 

In der Library können hierfür \textmono{store}-Systeme erstellt werden, welche die vorgegebenen Daten speichern und Methoden zur Veränderung der Werte bereitstellen. ARVal definiert hierfür zwei Systeme:

\begin{enumerate}
    \item \textmono{ObjectStore}: Dieses System speichert die Liste aller \textmono{Object}-Instanzen (vgl. Kapitel \ref{subsection:Kernsystem} Kernsystem: Application Layer [S. \pageref{subsection:Kernsystem}]) sowie das aktuell editierte Objekt. Zur Verwaltung der Objekte stellt das System die Methoden \textmono{addObject}, \textmono{removeObject} sowie \textmono{setEditingObject} bereit.
    \item \textmono{ThreeStore}: Dieses System speichert die Referenz zur aktuellen 3D-Szene. Die ist nötig, damit HTML-Elemente wie das Pop-up zum Hinzufügen neuer Objekte, auf Informationen der Szene zugreifen können.
\end{enumerate}

\subsection{Performance-Messung}

Zur Verbesserung und Bewertung der Performance des Validierungssystems wurde eine zweiteilige Performance-Analyse durchgeführt.

Im ersten Schritt wurde die Funktion des \textit{Flame Graph} im Chrome Browser sowie ein eigener Performance-Profiler genutzt, um die Laufzeit der einzelnen Komponenten und Engpässe des Systems zu ermitteln. Mittels dieser Analysen wurden mögliche Engpässe analysiert und ihre Laufzeit verbessert.

Ein Flame Graph ist eine Visualisierungsform zur Analyse von Laufzeitverhalten und Performance von Code. In dem Graph wird die Aufrufhierarchie von Funktionen oder Methoden als gestapelte, farblich kodierte Balken dargestellt. Die horizontale Breite eines Balkens entspricht der Laufzeit der jeweiligen Funktion inklusive ihrer Kindaufrufe. Funktionen, die häufig oder lange ausgeführt werden, erscheinen dadurch horizontal breiter. Die vertikale Anordnung zeigt die Aufrufstruktur: Eine Funktion, die von einer anderen aufgerufen wird, liegt direkt darunter. Flame Graphs zeigen auf einen Blick, welche Codestellen den größten Einfluss auf die Gesamtverarbeitung haben.

Zusätzlich wurde ein eigener Performance-Profiler \textmono{ValidationPerformance} erstellt, um die kumulierte Laufzeit der einzelnen Phasen über mehrere Durchläufe zu analysieren. Hierzu können einzelne Funktionen einen benannten Zeitabschnitt starten und beenden. Am Ende von jeweils 50 Durchläufen bietet der Profiler schließlich eine Übersicht über die Gesamtlaufzeit, durchschnittliche Laufzeit sowie Anzahl der Durchläufe der einzelnen benannten Anschnitte.
