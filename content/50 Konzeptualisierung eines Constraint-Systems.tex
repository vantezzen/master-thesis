\section{Konzeptualisierung eines Constraint-Systems}\label{section:Konzeptualisierung eines Constraint-Systems}

Das grundsätzliche Constraint-System soll zunächst auf den Erkenntnissen der vorangegangenen Forschungsprojekten aufbauen. 

Für eine native Code-Entwicklung für Webumgebungen wird konzeptioniert, dass der Code in TypeScript umgesetzt werden sollte. Hierdurch können objektorientierte Coding Patterns wie Interfaces und abstrakte Klassen genutzt werden, sowie eine Typisierung innerhalb des Codes ermöglicht. 

\subsection{Erstellung einer Syntax für Constraints}

Die Erstellung einer Syntax zur Definition von Regeln für Objekte wird in dieser Arbeit als \textit{Domain Specific Language} (DSL) Engineering Aufgabe betrachtet. DSL sind solche Programmiersprachen, welche auf spezifische Anwendungsfälle spezialisiert sind und hierdurch eine höhere Ausdruckskraft und Nutzungsfreundlichkeit in diesem Feld als generische Sprachen erreichen können. Im Gegensatz zu generischen Programmiersprachen muss das Ziel einer DSL nicht zwingend sein, einen ausführbaren Code zu generieren, sondern kann vielmehr auch der Ausführung eines größeren Systems dienen. \parencite[S. 316f]{mernikWhenHowDevelop2005}

Um ein strukturiertes Vorgehen bei der Ausarbeitung durchzuführen, orientiert sich dieses Kapitel an dem fünfstufigen Prozess zur DSL-Definition nach \textcite{mernikWhenHowDevelop2005}, mit Anpassungen zur besseren Integration in die bisherigen Vorarbeiten der vorherigen Kapitel.

\subsubsection{Entscheidung}

Der erste Schritt in der Entwicklung eines DSL nach \textcite{mernikWhenHowDevelop2005} ist die Entscheidung, ob ein DSL notwendig ist und um welchen DSL-Typ es sich handeln soll.

Der ARVal Constraint Syntax (nachfolgend \textit{ACS}) kann hierbei als “Data structure representation” typisiert werden, da es die Datenstruktur der Regelpakete als separate Datei darstellt.

Die Spezifizität dieser Art von Daten „relies on initialized data structures whose complexity may make them difficult to write and maintain. Such structures are often more easily expressed using a DSL“ \parencite[S. 323]{mernikWhenHowDevelop2005}. Aus diesem Grund wird die Implementierung eines spezifischeren DSL für den Regel-Syntax als sinvoll angesehen.

\subsubsection{Analyse}

Für die Konzeptionierung einer DSL wird empfohlen, zunächst eine Analyse der Problemdomäne durchzuführen. Teile dieser Ausarbeitung wurde in dieser Arbeit bereits in Kapitel \ref{section:Requirements-Engineering} als Teil des Requirements Engineering durchgeführt.

Abweichend zu \textcite{mernikWhenHowDevelop2005} werden zur besseren Einordnung der benötigten Komponenten der finalen Sprache werden zunächst formale Anforderungen an den Syntax definiert, analog zum bestehenden Requirements Engineering Ansatz.

Tabelle \ref{tab:syntax-requirements} stellt die aufgestellten Anforderungen an den Syntax dar.

R-1 stellt voraus, dass der Syntax leicht parse- und bearbeitbar ist. Die Definition einer neuen Syntax benötigt die Implementierung eines eigenen Text-Parsers, Validierers und, falls benötigt, Implementierung einer Sprachunterstützung für Code-Editoren.

Bei Nutzung einer bestehenden Syntax (z.B. JSON, XML etc.) kann das Text-Parsing durch bestehende Libraries geschehen und eine eigene Implementierung muss nur auf Ebene des Parsing und Validierung der Regel-Logik geschehen: „For an XML-based DSL, grammar is described using a [...] XML scheme [...] [P]arsing comes for free“ \parencite[S. 330]{mernikWhenHowDevelop2005}.

\medskip

Durch R-2 wird definiert, dass der Syntax erweiterbar für neue Elemente und Methoden sein soll. Werden beispielsweise in Zukunft neue Segmentierungsarten wie „Kreuzungsbereich“ genutzt, soll das neue Element ohne eine Neudefinition der Syntax nutzbar sein. Analog soll beispielsweise das Hinzufügen neuer Constraint-Methoden wie „nur Vertikal zu ...“ ebenfalls ohne Neudefinition ermöglicht werden.

\medskip

R-3 fügt hinzu, dass der Abstraktionsgrad angemessen für den Anwendungsbereich sein muss. Hierbei soll weder eine zu starke noch zu schwache Abstraktion stattfinden, um eine Sprache zu erhalten, die spezifisch für die Bürgerbeteiligung ist, in diesem Bereich jedoch generisch genutzt werden kann.

Speziell kann etwa von „Straßen“, „Grünflächen“ und „Häusern“ ausgegangen werden und es müssen nicht arbiträre Objekte in arbiträren Umfeldern unterstützt werden. Trotzdem soll, wie in R-2 definiert, eine einfache Erweiterung mit neuen Elementen möglich sein, ohne eine Neudefinition der Syntax.

Ebenfalls sollen Methoden generisch genug gehalten werden, um Dopplungen von Code zu vermeiden. Statt einer spezifischen Funktion “abstandZumGehweg()” sollte etwa stattdessen eine generisch nutzbare Funktion “abstandZu(gehweg)” erstellt werden.

\medskip

Durch R-4 wird definiert, dass eine Umsetzung als visueller Editor möglich sein soll. Zwar ist die Implementierung dieses Editors nicht im Kontext dieser Arbeit, jedoch soll dies zukünftig für die bessere Einbindung von Stadtplaner:innen geschehen und daher konzeptionell in der Syntax einfließen.

\medskip

R-5 fügt hinzu, dass die Regeln eindeutig (\textit{Non-ambiguous}) sein sollen. Hierdurch werden insbesondere Regeldefinitionen auf natürlicher Sprache bzw. durch Natural Language Processing (NLP) eingeschränkt.

Wie in Kapitel \ref{section:Verwandte Arbeiten} erarbeitet, sind Regeln in natürlicher Sprache oft abhängig von der individuellen Interpretation des Lesers. Beispielsweise bei einer Regel wie „Bänke dürfen nicht neben Fahrbahnen aufgestellt werden, wenn sie keine Barriere haben.“ kann rein vom Satzbau nicht gelesen werden, ob sich die Beschränkung „wenn Sie keine Barriere haben“ auf die Bänke oder die Fahrbahnen beziehen.

\textcite[S. 8]{sydoraRulebasedComplianceChecking2020} arbeitet auf, dass oft genutzte Begriffe in Regeln mehrdeutig interpretiert werden können. Eine Regel wie „Der Abstand zwischen zwei Objekten“ kann sich etwa auf den Abstand der Mittelpunkte der Objekte, Abstand der 2D-Grundfläche oder 3D Abstand des am nächsten gelegenen Punkt beziehen.

\medskip

Durch R-6 wird definiert, dass boolsche Operationen von UND- sowie ODER-Verknüpfungen konzeptionell mitgedacht werden sollen, inklusive einer Verschachtelung dieser Operatoren. Hierdurch sollen komplexere, kontextabhängige Regeln umgesetzt werden können wie „Steht die Bank an einem Gehweg, \textit{muss} diese parallel zum Weg stehen \textit{ODER} wenn die Bank auf einer Grünfläche steht, \textit{kann} diese beliebig rotiert werden“.

Wie bei der Umsetzung des Editors soll auch diese Funktion zunächst nur konzeptionell eingearbeitet werden, jedoch noch nicht in die Validierungseinheit implementiert werden. Aufgrund der Komplexität der Unterstützung arbiträr verschachtelter Elemente soll der Fokus dieser Arbeit primär auf die Implementierung und Integrierung der tatsächlichen Regeln liegen.

\medskip

Schließlich fügt R-7 hinzu, dass die Sprache sowohl positiv (\textit{Muss}) als auch negativ (\textit{Darf nicht}) formulierte Regeln unterstützen soll. Abhängig von der Regel kann es deutlich übersichtlicher sein, die Regel auf eine statt die andere Art zu definieren \parencite[S. 44]{trinhConstraintbasedApproachModelling2012} - beispielsweise „Eine Bank \textit{muss} auf einer Grünfläche stehen“ ist einfacher zu definieren als „Eine Bank \textit{darf nicht} stehen auf Straßen, Gehwegen, Schotterwegen, Kreuzungsbereichen, ...“.

\medskip

Durch die Einhaltung dieser formalen Anforderungen kann erhofft werden, die Argumentation einer finalen Syntax leichter und für den System passender umsetzen zu können.

\subsubsection{Design}

\epigraph{Design only what is necessary. Learn to recognize your tendency to over-design.}{\textcite[S. 327]{mernikWhenHowDevelop2005}}

Der dritte Schritt in der Syntax-Definition nach \textcite{mernikWhenHowDevelop2005} ist die Durchführung des Syntax-Designs. In diesem Schritt werden alle Vorüberlegen zur DSL genutzt, um einer Syntax zur Erfüllung dieser zu definieren.

\medskip

Zur Umsetzung von R-2 zur Erweiterbarkeit wurden bisherige Constraint-Systeme aus Kapitel \ref{section:Verwandte Arbeiten} und der Elicitation-Phase des Requirements Engineering untersucht.

Viele der Sprachen besitzen Tagging-ähnliche Systeme wie \textit{Object Properties} \parencite{sydoraRulebasedComplianceChecking2020}, \textit{Attachable Scripts} \parencite{vassionInvestigatingConstrainedObjects2023a} oder Klassen \parencite{xuMethodologyModelling3D2017} zur Identifizierung von Objekten nach bestimmten Eigenschaften.

Aus diesem Grund wird für \textit{ACS} ebenfalls ein Tagging-System als Grundlage der generischen Identifikation genutzt werden. Objekten sollen hierfür eine beliebige Anzahl an relevanten Tags hinzugefügt werden können, welche andere Objekte für die Regeldefinition nutzen können.

Beispielsweise können die Objekte „Straße“ und „Gehweg“ den Tag „asphaltiert“ besitzen. In den Regeln für den Objekttyp „Baum“ kann nun als Regel definiert werden, dass dieser nicht auf asphaltierten Untergründen platziert werden darf, ohne explizit alle möglichen Elementtypen nennen zu müssen.

Zukünftig können hierdurch beliebig neue Tags und Elemente hinzugefügt und können problemlos in bestehende Regeln eingearbeitet werden, ohne den Gesamtsyntax neu definieren zu müssen.

Die Nutzung eines Tagging-System ermöglicht zudem die Nutzung eines geeigneten Abstraktionsgrads (R-3). In Regeldefinitionen kann explizit von „Straße“, „Asphaltiert“ oder „Walkable“ ausgegangen werden, einzelne Methoden können jedoch trotzdem generisch auf alle Tag-Typen angewendet werden.

\medskip

Zur Erfüllung der Anforderung R-1 soll das Konzept der Language Specification genutzt werden, um die Infrastruktur einer bestehenden Host-Sprache zu nutzen und für die eigenen DSL zu spezifizieren. 

\textcite{mernikWhenHowDevelop2005} untersucht primär XML-basierte DSL. JSON bietet mittlerweile jedoch ebenfalls eine hohe Unterstützung zu bestehenden Werkzeugen an und besitzt mit JSON Schemas eine Möglichkeit, den Constraint-Syntax zu formalisieren und automatisch in bestehenden Code-Editoren validieren zu lassen \parencite{jsonschemaorganisationJSONSchema2025}.

Neben dem JSON-basierten Ansatz wurden zusätzlich andere Ansätze als SQL-ähnliche Statements, Code-ähnliche Methodenaufrufe oder OCL-ähnliche Aussagen untersucht, welche in Listing \ref{lst:schemaformate} [S. \pageref{lst:schemaformate}, \ref{anh:Schemaformate}] dargestellt sind. Die untersuchten Formate ergeben sich aus anderen Constraint-Sprachen, welche in den vorherigen Kapiteln untersucht wurden.

Inhaltlich besitzen alle untersuchten Formate ähnliche Elemente, damit die aufgestellten Anforderungen an der Syntax erfüllt werden. So besitzt der Syntax jeweils eine Möglichkeit, um das Tagging des Objekts vorzunehmen, die Transformation einzuschränken und die Platzierung zu validieren. 

Durch die Verbreitung von JSON im Umfeld von Webapplikationen, Ähnlichkeit zu nativen JavaScript-Objekten und Integration in bisherige Sprachen wurde dieser Syntax schließlich als Grundlage für \textit{ACS} genutzt. Die Nutzung des JSON-Schema-Systems kann ebenfalls konzeptionell für einen zukünftigen Regel-Editor (R-4) eingearbeitet werden. Die Regeln und Schemas können von diesem Editor verarbeitet und manipuliert werden, ohne eigene Parser-Systeme implementieren zu müssen.

\medskip

Die Umsetzung von R-5 (Eindeutige Regeln) wird durch die Nutzung eines festgelegten, schematisch formalisierten Syntax unterstützt, jedoch müssen die einzelnen Regeln ebenfalls darauf angepasst werden, eine eindeutige Identifikation zu ermöglichen.

\medskip

Für die Unterstützung von R-6 (Boolean Operationen) und R-7 (Positiver und Negativer Regeln) sollen zusätzlich einfache Verknüpfungen von Regeln und Definition einer Regel-Art (positiv/negativ) eingearbeitet werden. Die Integrierung von R-6 (Boolean Operationen) soll starke Inspiration von der Umsetzung in JSON Schema selber nehmen, durch die Nutzung von Subobjekten und Schlüsselwörtern wie „anyOf“.

\subsubsection{Schema-Definition}

Mithilfe dieser Vorüberlegungen kann schließlich ein finales Schema für \textit{ACS} definiert werden.

\begin{lstlisting}[frame=htrbl, caption={ACS Beispiel}, label={lst:acs-example}, language=php]
// Tree.constraints.json
{
	"$schema": "https://arval.vantezzen.io/schemas/acs.schema.json",
  "name": "Baum",
  "tags": ["tree", "plant", "nature"],

	"transform": [
		{ 
			"action": "allow-only",
			"subject": "rotation",
			"axes": ["y"],
			"active": false
		},
		// Same as
		{
			"action": "forbid",
			"subject": "rotation",
			"axes": ["x", "z"]
		},
		
    {
	    "action": "allow-only",
	    "subject": "scale",
	    "between": [0.5, 1.5] 
	  },
	],

  "placement": [
    {
	    "action": "forbid", 
	    "subject": "surface", 
	    "tags": ["asphalted", "water"]
	  },

    { "anyOf": [
        {
	        "action": "require",
	        "subject": "surface",
	        "tag": "planting_bed"
	      },
        {
	        "action": "require",
	        "subject": "distance",
	        "to": "road",
	        "min": 2.0
	      }
      ]
    }
  ]
}
\end{lstlisting}

Listing \ref{lst:acs-example} zeigt ein Beispiel für ein Regelpaket eines Baums in \textit{ACS}. Das JSON-Objekt enthält folgende Elements:

\begin{enumerate}
    \item \textmono{\$schema}: Über den Schema-Syntax kann Code-Editoren mitgeteilt werden, nach welchem JSON-Schema das Dokument geprüft werden soll. Über den Schlüssel \textmono{\$id} wird hierbei eine URL definiert, welche eine Konfiguration aller erlaubten Daten enthält.
    \item \textmono{name}: Der Name stellt einen lesbaren Namen für das Objekt dar und wird im späteren System insbesondere zur Erstellung lesbarer Fehlermeldungen wie „Der \textit{Baum} darf nicht auf dem \textit{Weg} gestellt werden“ genutzt.
    \item \textmono{tags}: Dieses Array definiert alle Tags, die diesem Objekt zugeordnet werden. Diese Liste sollte aus spezifischen (\textmono{tree}) bis generischen (\textmono{nature}) Tags zusammengesetzt werden.
    \item \textmono{transform}: Diese Liste definiert alle Regeln, welche bereits während der Bewegung des Objekts genutzt werden sollen. Diese Regeln sind für das AR-Basissystem von Bedeutung, da hierdurch die Transformation verändert wird.
    \begin{enumerate}
        \item \textbf{Regel 1 und 2}: Die erste und zweite Regel definiert, dass das Objekt nur um die y-Achse gedreht werden darf. Die Regeln sind in ihrer Aussage identisch, jedoch einmal als positive Regel (\textit{Lasse nur Rotation um die y-Achse zu}) und einmal als negative Regel (\textit{Verbiete alle Rotationen um die x- und z-Achse}) definiert - dies wird über das Keyword \textmono{action} definiert.
        Über die Property \textmono{subject} wird jeweils definiert, welche Transformations-Art eingeschränkt werden soll (Rotation, Skalierung oder Position). 
        Wie in User Story \textit{US-SP-005} definiert, existiert zusätzlich ein optionales Feld \textmono{active}, welches eine Regel temporär deaktivieren kann.
        Zusätzlich zu den Basisoptionen existieren pro \textmono{subject} eigene, weitere Optionen für Argumente, beispielsweise \textmono{axes} für die Einschränkung von Rotationen. 
        \item \textbf{Regel 3}: Die dritte Regel definiert zusätzlich eine Einschränkung der Skalierung zwischen 0,5x und 1,5x
    \end{enumerate}
    \item \textmono{placement}: Über das Array an placement-Regeln kann definiert werden, welche Platzierungen als valide angesehen werden sollen.
    \begin{enumerate}
        \item \textbf{Regel 1:} Die erste Regel verbietet die Platzierung des Baums auf Oberflächen mit dem Tag \textmono{asphalted} und \textmono{water}. Der Syntax zur Definition ist ähnlich zu Transformations-Regeln, besitzt jedoch andere \textmono{subject}-Optionen, etwa zur Oberfläche, anstatt sich auf Transformations-Achsen zu beschränken.
        \item \textmono{anyOf}: Anforderung R-6 (Unterstützung einfacher Boolean Operationen) wurde durch die Definition von Subobjekten integriert. Alle Regeln auf Rootebene müssen standardmäßig für eine valide Platzierung erfüllt werden (\textmono{AND}), komplexere Verbindungen müssen daher als separater Block definiert werden.
        Der Block enthält die Regeln „Erfordere die Platzierung auf dem Untergrund-Tag \textit{Pflanzbeet}“ und „Erfordere einen Abstand von 2 Meter zur nächsten Straße“. Hieraus entsteht die logische Verknüpfung „Der Baum muss einen Abstand von 2 Metern zur Straße besitzen - steht dieser jedoch in einem Pflanzbeet ist der Abstand nicht relevant (z.B. weil das Pflanzbeet die Wurzelbildung des Baums kontrolliert)“.
    \end{enumerate}
\end{enumerate}

Die Separierung von \textmono{transform}- und \textmono{placement}-Regeln wurde vorgenommen, da sich die beiden Regeltypen in unterschiedlichen Phasen befinden: während die \textmono{transform}-Regeln grundsätzlich und ohne Ausnahme schon bei der Verschiebung verhindert werden, werden \textmono{placement}-Regeln zunächst nur visuell dargestellt und werden erst beim Absenden eines Vorschlags als invalide blockiert.

Beispielsweise ist es grundsätzlich nicht sinnvoll, dass der Nutzer einen Baum kopfüber aufstellt (\textmono{transform}) und wird keine Situation geben, in der diese Bewegung sinnvoll ist, um einen validen Status zu erreichen. Auf der anderen Seite kann es sinnvoll sein, zuzulassen, dass der Baum zwischenzeitlich auf der Straße platziert wird, damit der Nutzer diesen auf die andere Straßenseite befördern kann (\textmono{placement}).

\medskip

Neben dem informellen Design durch ein konkretes Beispiel wurde ein formelles Design durch die Definition eines JSON-Schemas erstellt, erreichbar unter \url{https://arval.vantezzen.io/schemas/acs.schema.json}.

JSON-Schema Dateien erlauben die Definition von Attributen des Objekts (\textit{Properties}), einer Typisierung und Beschreibung der einzelnen Felder und Definition von verpflichtenden und freiwilligen Feldern.

Eine Besonderheit in der Definition der \textit{ACS}-Schema ist, dass Platzierungsregeln mehrere Ebenen tief verschachtelt sein können und daher rekursiv ausgewertet werden müssen. Zur Erfüllung der Anforderung R-6 (Boolean Operationen), sollen Verschachtelungen wie „\textmono{(a UND b) ODER (c UND (d ODER e))}“ ermöglicht werden. Zur Nutzung von boolischen Operatoren wurden die Elemente „anyOf“ (ODER) und „allOf“ (UND) hinzugefügt, inspiriert vom eigenen Syntax von JSON-Schema Definitionen.

\begin{lstlisting}[frame=htrbl, caption={Komplexe Verschachtelungen boolscher Operationen in ACS}, label={lst:acs-recursive}, language=php]
{
  "$schema": "./acs.schema.json",
  
  "placement": [
    {
      "anyOf": [
        {
          // (a OR b)
          "allOf": [a, b]
        },
        {
          // (c UND (d ODER e))
          "allOf": [
            c,
            // d ODER e
            {
              "anyOf": [d, e]
            }
          ]
        }
      ]
    }
  ]
}
\end{lstlisting}

Listing \ref{lst:acs-recursive} zeigt die Implementierung dieser Verschachtelung in \textit{ACS} über die Nutzung der „anyOf“ und „allOf“ Operatoren. Um diese Art von rekursiver Definition erlaubter Objekte in der JSON-Schema abzubilden, wurde die Unterstützung wiederverwendbarer Definitionen (\textmono{\$defs}) genutzt.

Intern wird durch die Nutzung der Definition \textmono{placementRuleOrGroup} erlaubt, dass die Platzierung von spezifischen Regeln oder weiteren booleschen Gruppen innerhalb der booleschen Gruppe möglich ist und kann so beliebig verschachtelt werden.

\subsection{Umgebungssegmentierung}

Während der Untersuchung verwandter Arbeiten in Kapitel \ref{subsection:Umgebungssegmentierung} [S. \pageref{subsection:Umgebungssegmentierung}] wurden vor allem zwei Methoden der Segmentierung herausgearbeitet.

Hierbei existiert die Segmentierung anhand geographischer Kartendaten. Dies erlaubt eine genaue 3D-Segmentierung von Straßen, Gebäuden, und anderen kartographierten Elementen. Zentrale Nachteile aus den untersuchten Arbeiten und für den Zweck dieser Arbeit sind jedoch vor allem die Ungenauigkeit der eigenen Standortdaten, sowie dass nicht-kartographierte Daten wie Bäume und Sitzbänke nicht segmentiert werden können.

Daneben existiert eine Segmentierung anhand der Bilddaten über Machine Learning Modelle. Vorteil ist vor allem eine hohe Genauigkeit der Daten, da diese direkt vom Gerät selber erfasst werden. Nachteile dieser Methode sind jedoch der hohe Rechenaufwand, Verzögerungen durch die Verarbeitung des Bildes, sowie, dass Ergebnisse nur auf der 2D-Plane des Kamerabild und nicht als 3D-Raum erkannt werden.

\medskip

Die Segmentierung der Elemente stellt in sich bereits ein breites Themenfeld dar. Aus diesem Grund soll für das zu entwickelnde System zunächst stattdessen ein statisches Segmentierungsmodell anhand vorgegebener Koordinaten umgesetzt werden.

Hierfür soll das System eine Konfigurationsdatei im GeoJSON-Format\footnote{\url{https://geojson.org/}} erhalten. GeoJSON stellt ein standartisiertes Format für die Darstellung von geographischen Daten in einer JSON-Datei bereit, über welches Flächen mittels verschiedener Geometrie-Typen - wie Polygone, Kreise oder Linien - sowie Informationen über die Art der Fläche - wie Straßen oder  Radwege - definiert werden. \parencite{butlerGeoJSONFormat2016}

Für die Entwicklung dieses Systems soll diese Definition zunächst als statische JSON Datei bereitgestellt werden, in zukünftigen Iterationen könnte dies jedoch über separate, dynamische Segmentierungs-Systeme geschehen. 

\subsection{Darstellung von Rückmeldungen}

Analog zur Segmentierung stellt auch die Darstellung von Rückmeldungen in sich bereits ein breites Forschungsfeld mit diversen Lösungsansätzen im Bereich der HCI dar \parencite{wangUsingVisualFeedback2023}. Im Rahmen der Elicitation anderer Software in Kapitel \ref{subsection:Elicitation} [S. \pageref{subsection:Elicitation}] wurden bereits verschiedene Visualisierungen untersucht.

Eine ausführliche Aufarbeitung und ein systematischer Vergleich aller Methoden im Kontext von AR-Feedbackdarstellung stellt ein  umfangreiches Forschungsfeld dar und wäre insbesondere im Rahmen empirischer Nutzerstudien zur Bewertung von Intuitivität und Verständlichkeit vielversprechend. Im Rahmen dieser Arbeit wird auf eine solche umfassende Untersuchung verzichtet, da sie den Umfang sprengen würde. Stattdessen orientiert sich die Umsetzung an den bewährten und in verwandten Systemen häufig eingesetzten Visualisierungsansätzen.

Zur Umsetzung oft identifizierter Visualisierungen in anderen Systemen sowie die Einbindung der Requirements an das System sollen vor allem drei Methoden zur Darstellung von Rückmeldungen genutzt werden:

Zunächst soll das Objekt bei Fehlplatzierung, ähnlich wie bei Software wie \textit{Super Mario Maker 2}, rot hinterlegt werden. Hierdurch kann zunächst ein generischer Fehlerstatus an den Nutzer gegeben werden, jedoch noch keine weiteren Informationen über die Ursache.

Aus diesem Grund soll zusätzlich eine Nachricht unterhalb des Objekts gezeigt werden, welches die genauere Fehlerbeschreibung darstellt. Die Fehlermeldungen sollen aus den Begründungsinformationen der Constraints generiert werden, um eine Nachvollziehbarkeit für Nutzer zu ermöglichen.

Zuletzt sollen die Fehlerbereiche selber ebenfalls rot hinterlegt werden. Ist beispielsweise die Platzierung auf einer Straße nicht erlaubt, so soll die Straße als ganzes rot hinterlegt werden. Hierdurch kann erhofft werden, dass die Bereiche verbotener Platzierung für Nutzer direkt ersichtlich werden und nicht erst erprobt werden müssen.

\subsection{Konzeption eines Administrationsinterface}
